معمارية الفصوص الثلاثة (Three-Lobe Architecture)

الأساس الهندسي

كل عقدة في الشبكة تعمل بثلاثة فصوص متوازية — كل فص يعمل مستقلاً ويتحدث مع الآخرين عبر حافلة بيانات داخلية (DMA ring). المبدأ: فصل الاستشعار عن التفكير عن التنفيذ.

📡
الفص الحسي
SENSORY LOBE — echowall
يسحب عينات I/Q الخام من LimeSDR عبر Zero-Copy DMA مباشرة إلى NPU. يحلل تشوه إشارة Wi-Fi (CSI) للكشف عن الحركة البشرية خلف الجدران.
▸ CSI Processing · 23ms latency · 470–698 MHz
🧠
الفص المعرفي
COGNITIVE LOBE — CasperEngine
يشغّل SIDSense للمسح الطيفي المستقل. يقرر: أي قناة TVWS فارغة؟ هل نقفز تردداً؟ هل الجار عقدة موثوقة؟ كل هذا بدون اتصال سحابي.
▸ SIDSense INT8 · δ-CRDTs · Kademlia DHT
🔐
الفص التنفيذي
EXECUTIVE LOBE — libp2p
يدير اتصالات الشبكة الشبكية (Mesh) ويغلّف كل رسالة بتشفير PQC (ML-KEM + ML-DSA). يسجّل كل خطأ في سجل hash-chained لا يقبل التعديل.
▸ ML-KEM · ML-DSA · AES-256-GCM · SQLite

Mesh Hop
لماذا Zero-Copy DMA مهم؟
في التصميم العادي: LimeSDR ← بيانات → ذاكرة CPU → يحوّلها CPU → ذاكرة NPU. هذا يرفع الاستهلاك لـ 40W+ ويسبب انصهاراً حرارياً. مع Zero-Copy: LimeSDR يكتب مباشرة في العنوان الذي ستقرأه NPU — CPU لا يلمس البيانات. النتيجة: 15W فقط. 
typedef struct __attribute__((packed)) {
    uint64_t source_phys_addr;  /* عنوان مخزن LimeSDR (Wi-Fi Radar)       */
    uint64_t dest_phys_addr;    /* مدخل echowall في الـ NPU               */
    uint32_t frame_len;         /* حجم الإطار — محاذي لـ 64-bit           */
    uint16_t channel_id;        /* تردد TVWS المستهدف (470–698 MHz)       */
    uint16_t dma_flags;         /* DMA_NEON | DMA_SVE | DMA_NO_ALLOC      */
} Niyah_DMA_Descriptor;

/* القيد الصارم: frame_len يجب أن يكون مضاعفاً لـ 64
   أي قيمة أخرى → kernel panic مقصود لمنع تلوث الذاكرة */
static_assert(sizeof(Niyah_DMA_Descriptor) == 20, "Struct layout changed!");
pub struct SovereignNeighborDiscovery {
    // Kademlia: يبحث عن الجيران في شبكة P2P محلية بدون DNS مركزي
    kademlia: libp2p::kad::Behaviour<libp2p::kad::store::MemoryStore>,

    // CoprimeScheduler: يجدوِل المسح الترددي بجدول أولي (coprime intervals)
    // لتقليل استهلاك الطاقة 90% — لا نمسح كل ثانية، بل كل N ثانية أولية
    low_power_scanner: CoprimeScheduler,

    // سجل hash-chained: كل حدث يحتوي SHA-256 للحدث السابق
    // مستحيل تعديله دون كسر السلسلة كلها
    audit_ledger: ImmutableHashChain,
}

impl SovereignNeighborDiscovery {
    pub fn discover(&mut self) -> Result<Vec<PeerId>, SovereignError> {
        // 1. مسح TVWS باستخدام SIDSense (بدون PAWS — بدون Google)
        let free_channels = self.low_power_scanner.scan_local_spectrum()?;
        // 2. إعلان وجودنا عبر Kademlia على أول قناة حرة
        self.kademlia.bootstrap()?;
        // 3. تسجيل الاكتشاف في السجل غير القابل للتعديل
        self.audit_ledger.append(Event::NeighborDiscovered)
    }
}

المرحلة الأولى: إثبات العقدة الواحدة (PoC)

Q4 2026

الهدف الوحيد لهذه المرحلة: تشغيل عقدة واحدة وإرسال رسالة مشفرة بين جهازين في غرفة واحدة. كل شيء آخر هو ترف. لا توسّع قبل أن يثبت النظام أنه يعمل في بيئة حقيقية.

خطوة 1 — إعداد البيئة (أسبوع 1)
  • تركيب HAVEN OS (fork مقسّى من Debian) على eMMC
  • تفعيل Secure Boot بمفتاح محلي — رفض أي kernel غير موقّع
  • تعطيل كل واجهات WiFi والبلوتوث الافتراضية — LimeSDR هو الراديو الوحيد
  • تثبيت ESP-IDF v6 وrust toolchain (armv8a-unknown-linux-gnu)
  • اختبار اتصال LimeSDR عبر USB3 — التحقق من firmware ECP5
خطوة 2 — Zero-Copy DMA (أسبوع 2–3)
  • كتابة kernel module لتسجيل منطقة DMA (64MB) في physical address space
  • تعديل LimeSuite driver لكتابة I/Q مباشرة في العنوان المسجّل
  • اختبار DMA descriptor: التحقق من عدم تدخل CPU (perf stat)
  • قياس درجة حرارة RK3588 تحت حمل كامل — يجب <75°C
  • التحقق من استهلاك الطاقة: يجب ≤15W بجهاز قياس حقيقي
خطوة 3 — echowall و SIDSense (أسبوع 3–4)
  • تحميل نموذج SIDSense INT8 على NPU عبر RKNN runtime
  • اختبار مسح TVWS: تحديد قنوات شاغلة في نطاق 470–698 MHz
  • تفعيل CSI extraction من I/Q samples
  • اختبار الكشف عن حركة خلف جدار خرساني (هدف: ≥94% دقة)
  • قياس الزمن من الإشارة إلى القرار: يجب ≤23ms
خطوة 4 — المكدس الشبكي (أسبوع 4–6)
  • تجميع libp2p مع Kademlia DHT (Rust، بدون tokio_tls خارجي)
  • تهيئة NXP SE051 وتوليد مفاتيح ML-KEM-768
  • إرسال أول رسالة مشفرة بين عقدتين — تحقق من decryption ناجح
  • اختبار الإجماع δ-CRDTs بين جهازين: تحقق من state sync بدون مركز
  • اختبار Phalanx Gate: حقن خطأ متعمد والتحقق من rollback آلي

المرحلة الثانية: شبكة الجيرة (100 عقدة)

2027 — 2028

الهدف: نشر أول "مشاع رقمي" في حي سكني بالرياض (مثل حطين أو الملقا). المؤشر الوحيد للنجاح: الجيران يتواصلون عبر الشبكة بدون إنترنت خارجي — مستدام تلقائياً.

تمثيل تخطيطي لنشر 100 عقدة في حيّين متجاورين — كل دائرة تغطي ~10 كم²

البروتوكول الفني — δ-CRDTs

المشكلة: الشبكة اللاسلكية تفقد حزم البيانات. إذا لم تصل رسالة، كيف تعرف عقدتان أنهما متزامنتان؟

الحل: δ-CRDTs (Delta Conflict-free Replicated Data Types). كل عقدة تحتفظ بحالتها المحلية. عند التقاء عقدتين، يتبادلان "دلتا" (التغييرات فقط، لا كل الحالة). القانون الرياضي يضمن أن أي ترتيب للتحديثات يصل إلى نفس النتيجة النهائية.

النتيجة: لا مركز → لا نقطة فشل واحدة → الشبكة تعمل حتى لو 30% من العقد أوفلاين.

LPI/LPD — التخفّي الإشاري

LPI (Low Probability of Intercept): إخفاء وجود الإشارة. LPD (Low Probability of Detection): التأكد من أن الإشارة تبدو كضجيج خلفية.

الطريقة: نبثّ الإشارة بقوة تحت مستوى ضجيج الخلفية الطبيعية (-101 dBm). الجهاز الخصم يرى "ضجيجاً" فقط. جهازنا يعرف مفتاح السبر الذي يفصل الإشارة عن الضجيج.

هذا للعمليات الحساسة فقط — الاتصال العادي لا يحتاج هذا المستوى.

  • طلب عروض أسعار من Crowd Supply وLimeMicro لـ 5 وحدات LimeSDR Mini 2.0

  • فتح repo خوارزم-kernel جديد مع CI/CD صارم (بدون شبكة خارجية)

الشهر 1 — التأسيس التقني

  • تركيب HAVEN OS على RK3588 مع Secure Boot
  • اختبار اتصال LimeSDR عبر USB3 — التحقق من firmware
  • كتابة أول kernel module لمنطقة DMA
  • اختبار استهلاك الطاقة بجهاز قياس فعلي (ليس تقديرياً)
  • البدء في خدمات PDPL الاستشارية لتوليد الإيرادات

الشهر 2–3 — البرمجيات

  • تشغيل SIDSense INT8 على NPU — قياس الدقة

  • تجميع libp2p مع Kademlia (Rust — offline)
  • تهيئة NXP SE051 وتوليد مفاتيح ML-KEM
  • اختبار أول تشفير/فك تشفير بين عقدتين

الشهر 4–6 — الإثبات
